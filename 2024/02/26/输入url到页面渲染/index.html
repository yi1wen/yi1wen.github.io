1. 浏览器检查是 url 还是搜索关键词，是 url 则下一步交给网络进程

2. 发起请求，查询缓存（协商缓存，强缓存）

强缓存：直接使用本地缓存支援，关心是否在有效期内

从本地缓存中查找也是需要发起请求，如果命中缓存则返回 200状态码和资源

缓存位置优先级：Service Worker > Memory Cache > Disk Cache > Push Cache

  • Serviec Worker：独立线程
  • Memory Cache：内存缓存（短期），忽略缓存控制头，直接读取，若禁止缓存则需要 Cache-Control: no-store。缓存资源有：通过 preloader 加载的资源（preloader 和 prefetch）并行预加载；preload 指令标记的资源。
  • Disk Cache：硬盘缓存（持久）
  • Push Cache：推送缓存，只在会话中存在

强缓存的 HTTP 头信息： Cache-Control > Empires

Empires：缓存过期时间，秒，绝对时间
Cache-Control：优先级高，max-age 表示在这个请求的返回时间的 max-age 时间内再次加载资源，就会命中强缓存

协商缓存：关心是否更新

304，让所缓存资源的星系发送给服务器，让服务器判断资源是否更新，是则返回 200 更新资源，无则返回 304，可直接用本地缓存的资源

Last-Modified/ If-Modified-Since：单位为秒，最后修改时间，当在秒以下的时间内修改 n 次则无法判断
Etag/ If-None-Match：资源特定版本的标识符，根据算法生成 hash 值，优先级更高，更精确。

协商缓存中，服务端判断资源是否更新是将请求头的信息与资源信息进行比对，如果匹配成功，则返回 304 状态码；否则，返回 200 状态码和新的资源。

浏览器刷新对缓存影响

  • F5 正常刷新，查找缓存
  • Ctrl+F5 强制重新加载，跳过缓存查找，直接从服务器加载资源。

3. DNS 查询

主要步骤

浏览器 DNS 缓存 -> 操作系统 DNS 缓存 -> 本地域名服务器缓存
向上级域名服务器进行迭代查询：从根域名服务器中获取顶级域名服务器地址，从顶级域名服务器中获取权限域名服务器地址，从权限域名服务器中获取目标 IP 地址
本地域名服务器，操作系统，浏览器都将 IP 地址缓存下来

DNS查询用 UDP 的原因

DNS 查询传输的内容少，短暂，要求速度快，服务器接收到的请求多负担大，还有如 TFTP 文件传输，SNMP 网络管理，NFS 远程文件服务器。

TCP

TCP 和 UDP 都处于传输层

TCP 应用场景为 SMTP 电子邮件，HTTP 万维网，FTP 文件传输

TCP 如何实现可靠传输：

  1. 重传机制：超时重传，丢包和确认迟到；选择性确认，对方一直确认同样的 ACK 确认报文，则重传这一段；超时达到一定次数后，会强制关闭连接
  2. 流量控制：滑动窗口（接收窗口和发送窗口），窗口中分为已发送未收到确认数据和未发送但在窗口内。滑动窗口实际存在与操作系统的内存缓冲区中；

窗口关闭（临近一个内存剩余值时就关闭，避免发送方窗口越来越小每次发送小数据造成不经济的传输）会发送窗口探测报文，接受发返回接收窗口大小
3. 网络拥塞控制（发送方窗口）:
慢开始：指数增长，每收到一个 ACK，拥塞窗口 + 1，一直到慢启动门限
拥塞避免：线性增长，传一组才+1，也就是每个ACK，+ 1/拥塞窗口；
快重传：当出现丢包后（3个重复的ACK）就要快恢复，不必等待超时
快恢复：更新慢启动门限和拥塞窗口为当前拥塞窗口的一半，再继续拥塞避免

发送请求：建立 TCP 连接

[https://cloud.tencent.com/developer/article/2320329?from_column=20421&from=20421]

三次握手

  1. 第一次：客户端发送 SYN=1，序列号 seq=x 的包到服务端，客户端处于 SYN-SENT 同步已发送状态：表示“我请求和你建立连接”，服务端接收到后知道客户端发送能力，服务端的接收能力正常
  2. 第二次：服务端发送 SYN=1，ACK=1,序列号 seq=y，确认号 ack=x+1 的确认响应包到客户端，服务端处于 SYN-RCVD 同步收到状态：表示“我收到了你的请求，我也请求和你建立连接”，客户端接收到后知道服务端的发送，接收能力正常，客户端的发送，接收能力也正常。
  3. 第三次：客户端发送 ACK=1，seq=x+1 的确认报文给服务端，客户端进入 ESTABLISHED 已建立连接状态：表示“我也收到了你的请求”，此时服务端也已知服务端的发送接收能力和客户端的发送接收能力正常。

为什么需要第三次握手

  1. 第二次握手后，客户端已知自己和服务端的发送，接收能力正常，但是服务端仅仅知道 客户端的发送能力正常和自己的接收能力正常，还需要知道客户端的接收能力和自己的发送能力正常
  2. 防止由于网络阻塞原因，重复发送的连接请求，延迟到达，而造成错误的再次建立连接.
  3. 按照 SYN=1，和 ACK=1 的标志符来看，客户端和服务端双方都要请求连接一次，收到对方确认一次，第一次握手是客户端请求，第二次握手是服务端请求，且确认客户端的请求收到，第三次握手是确认服务端的请求收到
