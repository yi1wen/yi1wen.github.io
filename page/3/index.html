避坑2：Vue2父组件异步传递props值，子组件接收不到

发表于 2024-04-25

现象

情况

  • 在父组件 created 中，发送请求
  • 父组件传值给子组件请求回来的数据
  • 子组件 created 中，使用这个数据

问题

  • 父组件请求回来的数据有快慢，导致子组件created中使用这个数据的时候，有时有，有时无

原因

  • 父created - 父beforeCMount - 子created - 子beforeMount - 子mounted
  • 异步数据返回时间不定
  • async ，await ，不阻塞，直接跳到下一个生命 周期函数

解决方法

  1. v-if 子组件，等父组件数据传回来时再进行渲染
  2. 子组件 watch 中，监听 props 值，加工赋值到自己的data数据中，若 watch 中加工可能会出现页面组件延迟改变的抖动

避坑1：页面缩放时scrollTop是小数

发表于 2024-04-25

现象

  • 监听滚动窗口位置，当滚动到元素底部时，请求下一组数据并显示，会出现位置判断不准确，无法加载数据的情况

js this.scrollHeight - this.scrollTop <= this.clientHeight // 滚动到底部但是判断为 false，差了一点点

原因

  • 电脑高分辨率，使得可以由浮点数来表示坐标
  • 缩放后分辨率有比例的转化，转化成浮点数

解决

考虑小数，四舍五入，或为了提高用户体验，改为距离底部有一段距离时，就可请求数据

内存泄漏

发表于 2024-03-14

内存泄露

占着内存但没有用

容易出现的场景

  1. 意外的全局变量
  2. 闭包，定时器回调引用外部的变量
  3. 缓存未释放：尤其是dom元素的缓存，移除dom元素后还存在于内存中；用对象做缓存；模块机制的缓存；队列的出不敷入（队列拥塞）
  4. 缓存重复：进程间不能共享缓存造成缓存重复

如何解决

  1. 手动赋值 null 释放，delete 释放
  2. 停止定时器
  3. 适当使用 weakmap，weakset
  4. use strict 解决意外全局变量
  5. Performance 记录快照,监测有无内存泄露
  6. 避免缓存过多：限制对象键值，LRU 算法，清空队列方法（监控长度）；
  7. 避免缓存重复：进程外缓存，可跨进程共享、
  8. 对于队列：异步调用的超时模式，拥塞时的拒绝模式

TCP

发表于 2024-03-09

1. TCP 长连接、心跳机制、轮询

TCP 长连接：保持连接，直到显式关闭连接或发生连接中断
心跳机制：因为长连接保持连接，所以可以用心跳机制，一在连接空闲时发送数据，防止超时关闭连接；二作为探测消息，检测连接状态
轮询：查询是否有新数据，关心是否有新数据，存在延迟和带宽浪费，心跳机制更轻量

2. TCP 长连接与 Websocket

  • TCP 是半双工通信，双方交替发送和接受数据；Websocket 全双工同行，同时发送和接受数据
  • Websocket 适用于对实时性要求高的场景，实时双向通信；如果只是保持连接，不需要双向的话，TCP 长连接更适用

TCP

TCP 和 UDP 都处于传输层

TCP 应用场景为 SMTP 电子邮件，HTTP 万维网，FTP 文件传输

TCP 如何实现可靠传输

  1. 超时重传：丢包和确认迟到；选择性确认，对方一直确认同样的 ACK 确认报文，则重传这一段；超时达到一定次数后，会强制关闭连接

  2. 流量控制：滑动窗口（接收窗口和发送窗口），窗口中分为已发送未收到确认数据和未发送但在窗口内。滑动窗口实际存在与操作系统的内存缓冲区中；

窗口关闭（临近一个内存剩余值时就关闭，避免发送方窗口越来越小每次发送小数据造成不经济的传输）会发送窗口探测报文，接受发返回接收窗口大小
3. 网络拥塞控制（发送方窗口）:
慢开始：指数增长，每收到一个 ACK，拥塞窗口 + 1，一直到慢启动门限
拥塞避免：线性增长，传一组才+1，也就是每个ACK，+ 1/拥塞窗口；
快重传：当出现丢包后（3个重复的ACK）就要快恢复，不必等待超时
快恢复：更新慢启动门限和拥塞窗口为当前拥塞窗口的一半，再继续拥塞避免
4. 确认应答机制，发送 ack 报文
5. 确认序列号，对失序报文进行重排（滑动窗口）
6. 丢弃重复数据包，防止数据冗余重复
7. 数据包校验：防止发送的是错误数据

发送请求：建立 TCP 连接

[https://cloud.tencent.com/developer/article/2320329?from_column=20421&from=20421]

三次握手

  1. 第一次：客户端发送 SYN=1，序列号 seq=x 的包到服务端，客户端处于 SYN-SENT 同步已发送状态：表示“我请求和你建立连接”，服务端接收到后知道客户端发送能力，服务端的接收能力正常
  2. 第二次：服务端发送 SYN=1，ACK=1,序列号 seq=y，确认号 ack=x+1 的确认响应包到客户端，服务端处于 SYN-RCVD 同步收到状态：表示“我收到了你的请求，我也请求和你建立连接”，客户端接收到后知道服务端的发送，接收能力正常，客户端的发送，接收能力也正常。
  3. 第三次：客户端发送 ACK=1，seq=x+1 的确认报文给服务端，客户端进入 ESTABLISHED 已建立连接状态：表示“我也收到了你的请求”，此时服务端也已知服务端的发送接收能力和客户端的发送接收能力正常。

为什么需要第三次握手

  1. 第二次握手后，客户端已知自己和服务端的发送，接收能力正常，但是服务端仅仅知道 客户端的发送能力正常和自己的接收能力正常，还需要知道客户端的接收能力和自己的发送能力正常
  2. 防止由于网络阻塞原因，重复发送的连接请求，延迟到达，而造成错误的再次建立连接.
  3. 按照 SYN=1，和 ACK=1 的标志符来看，客户端和服务端双方都要请求连接一次，收到对方确认一次，第一次握手是客户端请求，第二次握手是服务端请求，且确认客户端的请求收到，第三次握手是确认服务端的请求收到

四次挥手

一次 FIN 请求，需要一次确认 ACK

  1. 第一次： 用户端发送标识符 FIN=1，序列号 seq=u，表示：我的数据发送完了，我想和你断开连接
  2. 第二次：服务端确认
  3. 第三次：服务端发送完数据后，发送关闭请求，同第一次
  4. 第四次：客户端确认

为什么四次

半关闭状态，可接受不可发送
服务端还有数据要发送，客户端没有了

为什么等待 2MSL

最大往返时间，防止服务端没有收到 ACK 报文后重传

TCP 和 UDP 的区别

  1. TCP 是面向连接的， UDP 是无连接的
  2. TCP 支持点对点，UDP支持一对一，一对多，多对多
  3. TCP 是可靠的，UDP 是不可靠的
  4. TCP 拥塞控制机制，UDP 没有
  5. TCP 面向数据流，UCP 面向数据报（固定长度）
  6. TCP 有缓冲区，UDP 只有接受缓冲区

状态码

发表于 2024-02-29

1开头

  • 101 切换协议，http到websocket

缓存

304 协商缓存

重定向

  • 301（GET和HEAD），308（请求method不变，POST） 永久重定向，301最好用于GET或HEAD方法的请求，308用于POST
  • 302（GET和HEAD），303（PUT和POST的响应，重定向指向的链接不是新上传的资源，而是另外一个页面，比如消息确认页面或上传进度页面，用GET），307（请求method和body不变） 临时重定向

2开头 请求成功

  • 200 请求成功：除了从服务器处可以获得，查询缓存如果命中也会返回200
  • 204 请求成功，但客户端客户不需要离开当前页面，并不是内容为空，浏览器在收到204状态码后，应该继续发送后续请求以获取资源的实际内容。：使用惯例是，在 PUT 请求中进行资源更新，但是不需要改变当前展示给用户的页面，那么返回 204 No Content。如果创建了资源，则返回 201 Created 。如果应将页面更改为新更新的页面，则应改用 200 。
  • 206 请求一部分资源，成功返回一范围资源：Content-Range，涉及http1.1的分块传输编码，分片上传

错误

  • 401，缺乏身份验证凭证， WWW-Authenticate 包含如何验证的信息，可以重新验证
  • 403，禁止访问，服务端有能力处理请求，但客户端没有权限，重新验证也不会改变这个状态
  • 416，所请求的范围无法满足，发生在断点续传，Content-Range
  • 412，先决条件失败，除GET，HEAD之外的方法进行条件请求时，由首部字段If-Unmodified-Since 或 If-None-Match 规定的先决条件不成立的情况下，通常是上传或修改文件

垃圾回收机制

发表于 2024-02-29

总结

  1. 标记清除
  2. 标记整理
  3. 引用计数
  4. 分代式
  5. 全停顿
  • 如何处理垃圾回收后内存碎片化：标记整理（将存活对象集中到一端，编辑之外的区域清除）

标记清除

将每个变量都打上清除标记，若存在于上下文当中的变量，则去除清除标记，最后将剩下拥有清除标记的变量回收释放内存，需要随票

标记整理

将活着的内存向一端移动，覆盖死内存位置，移动后直接清理掉边界外的内存

引用计数

变量被引用的次数，我被引用的次数加一，若次数为零，则释放变量，回收内存

分代式

  • 新生代：新生代是生命周期短，新创建的变量，新生代垃圾回收频率高，用高效的垃圾回收算法

Scavenge算法：复制活着的对象 + 分空间。新生代空间分为 From 空间（正使用空间）和 To 空间（闲置空间），将存活对象复制从 From 到 To，复制前检查能否晋升到老生代
对象晋升条件：是否已经经历过 Scavenge 算法；To 空间占比超过 25%

  • 老生代，老生代是生命周期长，持久存在的变量，老生代垃圾回收频率低，用成熟稳定的垃圾回收算法

不再适用 Scavenge 算法，存活多，分区浪费空间，老生代用标记清除

全停顿

垃圾清除需要的时间，暂停应用逻辑
改进方法：增量标记、整理：垃圾回收与应用逻辑交替执行
